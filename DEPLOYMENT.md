## Project Deployment & Overview Guide

This document explains how to understand, build, and deploy the **ValintFoods** project to your remote server.

---

## 1. Project Overview

- **Name**: `ValintFoods`  
- **Purpose**: Marketing / company website for ValintFoods, built as a static React SPA.  
- **Build output**: Static files generated into the `dist/` directory.  
- **Primary deployment target**: Custom hosting (e.g. your WordPress webspace), by uploading `dist/` contents.

---

## 2. Tech Stack

- **Framework**: React 19  
- **Language**: TypeScript  
- **Build Tool**: Vite 5  
- **Routing**: React Router DOM 7  
- **Maps**: Leaflet + React Leaflet  

Key NPM scripts (from `package.json`):

```json
{
  "scripts": {
    "dev": "vite",
    "build": "tsc && vite build",
    "preview": "vite preview"
  }
}
```

│   ├── Map.tsx        # Leaflet map integration
│   ├── ScrollReveal.tsx
│   └── ScrollToTop.tsx
├── pages/             # Page-level components (route targets)
│   ├── Home.tsx / Home.module.css
│   ├── About.tsx
│   ├── Product.tsx / Product.module.css
│   └── Contact.tsx
├── data/
│   └── products.ts    # Product groups & product metadata
├── styles/
│   ├── App.module.css # Global layout (app shell, main padding/margins)
│   └── global.css     # Global resets / typography / base styles
├── utils/
│   ├── assets.ts
│   └── productAssets.ts
└── style.css          # Vite root-level style hook (if used)
```

High-level responsibilities:

- **`App.tsx`**: Wraps the app in the header (`Navbar`), sidebar, and `<main>` area for routed pages.  
- **`Home.tsx`**: Landing page, mainly shows `Carousel` and main sections.  
- **`Product.tsx`**: Product page that reads `productGroups` from `src/data/products.ts` and renders categorized product panels and product navigation.  
- **`Contact.tsx`**: Contact page with form and map.  
- **`Navbar.tsx`**: Fixed header with navigation and product group dropdown.  
- **`Carousel.tsx`**: Full-screen hero carousel at the top of the home page.

### 5.2 (Optional) Inspect the Remote `wordpress` Directory

```

Once logged in:

```bash
cd ~/wordpress
ls -lah
```

This lets you verify what’s already there before overwriting anything.

### 5.3 Upload the Build to `~/wordpress/`

From your local machine (not inside ssh), run:

```bash
cd /Users/jader/jader-s.github.io/jader-s.github.io

scp -r dist/* a1478919@access-5018136900.webspace-host.com:~/wordpress/
```

Explanation:

- **`scp`**: Secure copy over SSH.  
- **`-r`**: Recursively copy directories.  
- **`dist/*`**: All files and folders inside `dist/` (but not the `dist` folder itself).  
- **`~/wordpress/`**: Target directory on the remote server.

This will overwrite existing files in `~/wordpress/` with the same names (e.g. `index.html`).

### 5.4 (Optional) Back Up Existing Content

Before overwriting, you can create a backup of the current `wordpress` directory:

```bash
ssh a1478919@access-5018136900.webspace-host.com 'cd ~ && cp -r wordpress wordpress_backup_$(date +%Y%m%d_%H%M%S)'
```

Then upload the new build as in step 5.3.

### 5.5 (Optional) Clean the Target Directory

If `~/wordpress/` is used **only** for this static site and you want a clean deployment:

```bash
ssh a1478919@access-5018136900.webspace-host.com 'rm -rf ~/wordpress/*'
```

Then:

```bash
scp -r dist/* a1478919@access-5018136900.webspace-host.com:~/wordpress/
```

> **Warning**: `rm -rf ~/wordpress/*` will delete everything inside `~/wordpress/`. Only do this if you are sure you don’t need the existing content.

## 6. Notes on Running Under WordPress Hosting

Even though the path is named `wordpress`, in many hosting environments this is just a directory under your account. This project is a **pure static front-end**:

- No PHP is required for the React app itself.  
- The presence of `404.html`, `_headers`, and `.htaccess` in `dist/` can help with:
  - SPA-style routing (serving `index.html` for unknown routes, depending on server rules).  
  - Caching and compression (if the host honors these files).

If your provider is running Apache or Nginx with PHP (typical WordPress hosting), your static assets in `~/wordpress/` should still be served normally, as long as:

- The document root is pointing to this directory (or a subdirectory).  
- There are no conflicting rewrite rules that prevent loading `index.html` directly.

If you experience routing issues (e.g. `/product` returns 404 on refresh), check:

- That `index.html` is served as the default document.  
- That `.htaccess` from `dist/` is actually applied by the hosting environment.
